Apollo is built around three main principles: **modularity**, **event-driven concurrency**, and **real-time transparency**. The backend is a .NET 9 ASP.NET Core Web API that orchestrates a team of AI agents. A React frontend connects to it over SignalR to stream live research progress to the user. PostgreSQL with the pgvector extension acts as the persistent store for both relational data and vector embeddings.

## Project modules

Apollo is organized as a multi-project .NET solution. Each project has a single, well-defined responsibility.

| Project | Responsibility |
|---|---|
| `Apollo.Api` | ASP.NET Core Web API — HTTP endpoints, SignalR hubs, DI wiring, background service registration |
| `Apollo.Agents` | All AI agent logic — orchestration, plugins, state management, memory, event queues |
| `Apollo.Data` | Entity Framework Core models, `ApolloDbContext`, database migrations, repository access |
| `Apollo.Client` | React (Vite) frontend — TanStack Router & Query, SignalR client, UI components |
| `Apollo.Config` | Centralized configuration — API keys, model deployment names, database connection strings |
| `Apollo.Search` | Exa AI search integration — `ISearchService`, `WebSearchRequest`/`WebSearchResponse` models |
| `Apollo.Notifications` | Email notifications — `IEmailService`, research-complete email templates |

## Event-driven design

Apollo avoids blocking the HTTP request thread by routing long-running work through .NET channels. There are two independent event buses.

### Research events bus

When a user saves a research plan, `ResearchEventHandler.HandleResearchStart` writes a `ResearchStartEvent` to the `ResearchEventsQueue` channel. The `ResearchProcessor` background service (`BackgroundService`) continuously reads from that channel and spawns a scoped `ResearchOrchestrator` for each event.

```csharp ResearchEventsQueue.cs
public class ResearchEventsQueue : IResearchEventsQueue
{
    private readonly Channel<ResearchStartEvent> _channel;

    public ResearchEventsQueue()
    {
        _channel = Channel.CreateUnbounded<ResearchStartEvent>(
            new UnboundedChannelOptions { SingleReader = true }
        );
    }

    public ChannelReader<ResearchStartEvent> Reader => _channel.Reader;
    public ChannelWriter<ResearchStartEvent> Writer => _channel.Writer;
}
```

### Ingest events bus

While agents are running, Athena writes batches of `WebSearchResult` objects to the `IngestEventsQueue`. The `IngestProcessor` background service drains that queue and calls `IngestEventHandler.HandleIngestAsync`, which parallelizes the Kernel Memory ingestion calls so that web crawling never blocks the agent chat loop.

## SignalR real-time updates

The `IClientUpdateCallback` abstraction (injected into orchestrators, plugins, and the state manager) wraps the SignalR hub. Every component that needs to push a message to the browser calls one of its methods:

| Method | Payload type | When emitted |
|---|---|---|
| `StreamPlannerResponse` | `string` chunk | Streaming tokens from the planning chat |
| `SendResearchFeedUpdate` | `ResearchFeedUpdateEvent` | Progress messages, search queries, search results, snippets, TOC |
| `SendAgentChatMessage` | `AgentChatMessageEvent` | Each agent's streamed chat turn |
| `SendTimelineUpdate` | `TimelineUpdateEvent` | Question status changes (Pending → InProgress → Completed) |

The React client connects to the SignalR hub on page load and subscribes to all four event types, rendering them in real time without any manual polling.

## Deployment

The backend ships as a Dockerfile for deployment to platforms such as Render.com or Azure App Service. The React frontend is built as a static bundle (`bun run build`) and hosted separately on Vercel or Netlify, with the `VITE_API_URL` environment variable pointing to the deployed API.

<Note>
  Apollo's CORS policy must explicitly allow the deployed client origin. Set the allowed origins in `Apollo.Api` before deploying to production.
</Note>
